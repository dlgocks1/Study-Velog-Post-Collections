## 2.1 네 개의 영역

DDD에서는 복잡한 비즈니스 로직이 기술적인 코드와 뒤섞이지 않도록 영역을 엄격히 나눔. 이를 통해 비즈니스 핵심인 도메인이 외부 환경(DB, UI 등)이 변해도 흔들리지 않게 보호함.

1. 표현 영역 (Presentation Layer)
설명: 사용자의 요청을 받아 응용 영역에 전달하고, 처리 결과를 사용자에게 보여주는 계층임. HTTP 요청 변환이나 JSON 응답 생성이 여기서 일어남.

2. 응용 영역 (Application Layer)
설명: 시스템이 사용자에게 제공해야 할 **'기능'**을 실행함. 도메인 모델들을 조합하여 비즈니스 로직이 흐르도록 순서를 제어하며, 트랜잭션 관리도 이 계층에서 담당함.

3. 도메인 영역 (Domain Layer)
설명: 비즈니스의 핵심 규칙(Entity, VO)과 로직이 살아있는 가장 중요한 계층임. 외부 세상이 어떻게 돌아가는지(DB가 무엇인지 등) 전혀 몰라야 하며, 오직 비즈니스 문제 해결에만 집중함.

4. 인프라스트럭처 영역 (Infrastructure Layer)
설명: 구현의 세부 사항을 담당함. DB 연동, 메시지 큐 송신, 외부 API 호출 등 실제 기술적인 구현체가 이 계층에 위치함.

## 2.2 계층 구조 아키텍처

![](https://velog.velcdn.com/images/cksgodl/post/fe7fdd2a-ce7f-42f5-9f32-90e562934db5/image.png)

응용 계층과 도메인 계층이 인프라스트럭처(DB, 외부 API 등)에 직접 의존하게 되면, 기술이 바뀔 때마다 핵심 비즈니스 로직을 수정해야 하는 위험이 생김. 이를 해결하기 위해 DDD에서는 **의존성 역전 원칙(Dependency Inversion Principle, DIP)**을 활용함.


## 2.3 DIP

DIP는 객체 지향 설계의 5대 원칙(SOLID) 중 하나로, 하위 수준 모듈의 변경이 상위 수준 모듈에 영향을 미치지 않도록 의존 방향을 설정하는 핵심 전략이다.

- DIP (의존성 역전 원칙)의 본질
기본적인 설계에서는 상위 계층이 하위 계층에 의존함. 하지만 DIP를 적용하면 **"상위 계층과 하위 계층 모두가 추상화(인터페이스)에 의존"**하게 되어 의존성이 역전됨.

### DIP 적용 전 vs 후 비교

❌ DIP 적용 전 (직접 의존)
상위 계층이 하위 계층의 구체적인 클래스를 직접 알고 있는 상태임.

문제점: DB를 MySQL에서 MongoDB로 바꾸면 상위 계층인 OrderService 코드도 전부 고쳐야 함.

✅ DIP 적용 후 (의존성 역전)
상위 계층(도메인)이 인터페이스를 정의하고, 하위 계층(인프라)이 이를 구현함.

장점: 상위 계층은 인터페이스만 바라보므로 하위의 구체적인 변경에 영향을 받지 않음.

```kotlin
// 1. [도메인 계층] - 추상화에 의존함 (상위 모듈)
interface MessageSender {
    fun send(message: String)
}

class NotificationService(private val sender: MessageSender) {
    fun notify(msg: String) {
        sender.send(msg) // 구체적인 방식은 모르지만 보낼 수 있음
    }
}

// 2. [인프라 계층] - 추상화를 구현함 (하위 모듈)
class EmailSender : MessageSender {
    override fun send(message: String) {
        println("이메일 발송: $message")
    }
}

class SmsSender : MessageSender {
    override fun send(message: String) {
        println("SMS 발송: $message")
    }
}
```

> DIP는 **"저수준의 기술적 세부 사항이 고수준의 비즈니스 결정에 영향을 주지 못하게 하는 것"**

![](https://velog.velcdn.com/images/cksgodl/post/05059c32-d093-40c9-a40b-6140fa057d0f/image.png)

### DDD와 OOP의 관련성?

객체지향 프로그래밍(OOP)과 DDD(도메인 주도 설계)의 관계는 매우 깊음. 한마디로 정의하자면 **"DDD는 OOP의 원칙을 비즈니스 영역에 가장 극대화하여 적용한 설계 철학"**이라고 할 수 있음.

## 2.4 도메인 영역의 주요 구성요소

### 🍕 배달앱으로 이해하는 DDD 구성요소

#### 1. 엔티티 (Entity) — "누구인지"가 중요한 놈

배달앱에서 **주문(Order)**을 생각. 주문 #1234와 주문 #1235는 내용이 똑같아도 다른 주문이다. 왜? 고유 ID가 다르니까.

그리고 주문은 상태가 변한다. 접수 → 조리중 → 배달중 → 완료. 이렇게 라이프사이클을 가지면서도, 주문 #1234는 처음부터 끝까지 주문 #1234다. 이게 "연속성"과 "정체성"이라는 뜻이다.
```kotlin
// 엔티티: ID로 구분, 상태가 변함
class Order(
    val id: Long,           // ← 이게 정체성
    var status: OrderStatus  // ← 이게 변해도 같은 주문
)
```

#### 2. 밸류 객체 (Value Object) — "값 자체"가 중요한 놈
배달 주소 "서울시 강남구 테헤란로 123"은 누가 쓰든 같은 주소. ID가 필요 없이 값이 같으면 같은 것.

```kotlin
// VO: ID 없음, 값이 같으면 동일
data class Address(
    val city: String,
    val street: String,
    val zipCode: String
)
```

#### 3. 애그리거트 (Aggregate) — "한 묶음으로 같이 변경해야 하는 덩어리"

핵심은 **트랜잭션 경계** 주문의 경우를 생각. 주문 하나에는 이런 것들이 딸려 있음

```
주문(Order) ─── 루트 엔티티 (대장)
  ├── 주문항목(OrderLine) ─── VO (피자 2판, 콜라 1개)
  ├── 주문자(Orderer) ─── VO (이름, 연락처)
  └── 배송정보(ShippingInfo) ─── VO (주소, 요청사항)

// ✅ 독립적으로 변경되는 놈들을 별도 애그리거트로 분리
[Order 애그리거트]        [Payment 애그리거트]     [Delivery 애그리거트]
Order ─── 루트           Payment ─── 루트         Delivery ─── 루트
  ├── OrderLine            ├── PaymentMethod        ├── DeliveryStatus
  ├── Orderer              └── PaymentHistory       └── DriverInfo
  └── ShippingInfo
```

이것들은 항상 같이 저장되고, 같이 변경되어야 한다. "피자 2판 주문했는데 DB에는 주문만 저장되고 주문항목은 안 들어갔어요" 이러면 안됨

그래서 이 묶음 전체를 하나의 단위로 관리하는 게 애그리거트이고, **Order가 이 묶음의 대장(Aggregate Root)**이 돼서 외부에서는 반드시 Order를 통해서만 내부 데이터를 변경할 수 있다.

```kotlin
// ❌ 이렇게 직접 접근하면 안 됨
orderLine.changeQuantity(3)

// ✅ 반드시 루트를 통해서
order.changeOrderLineQuantity(orderLineId, 3)

```

그렇다면 매 조회마다 `Order`을 조회해야한다.

```kotlin
// ✅ 원칙대로라면 이렇게
val order = orderRepository.findById(orderId)
order.changeShippingInfo(newAddress, newRequest)
orderRepository.save(order)

class Order {
    fun changeShippingInfo(newInfo: ShippingInfo) {
        // 문지기 역할: 비즈니스 규칙 검증
        if (status != OrderStatus.PAYMENT_COMPLETED) {
            throw IllegalStateException("배송지 변경은 결제완료 상태에서만 가능")
        }
        if (status == OrderStatus.DELIVERING) {
            throw IllegalStateException("이미 배달 출발했는데 주소를 바꿔?")
        }
        this.shippingInfo = newInfo  // 검증 통과 후에만 변경
    }
}
```

> "매번 통째로 조회하면 비효율 아니야?"

따라서 현실에서는 몇 가지 방법으로 해결한다.

**1) 애그리거트를 작게 설계한다**

Order 안에 결제 이력, 리뷰, 배달 추적까지 다 넣으면 무거워지기 때문에, 그래서 애초에 애그리거트 경계를 잘 나누는 게 핵심이다. **"같이 변경되어야 하는 최소 단위"만 묶는 것**

```
// ❌ 너무 큰 애그리거트
Order ─── 주문항목, 배송정보, 결제이력, 리뷰, 배달추적, 쿠폰...

// ✅ 적절하게 분리
Order ─── 주문항목, 배송정보, 주문자
Payment ─── 별도 애그리거트
Delivery ─── 별도 애그리거트
Review ─── 별도 애그리거트
```

이렇게 하면 배송정보 바꿀 때 Order만 조회하면 되니까 가벼움.

**2) CQRS 패턴으로 조회와 변경을 분리한다**

- 변경(Command)은 애그리거트 루트를 통해 원칙대로 하되, 
- 단순 조회(Query)는 애그리거트를 무시하고 바로 DB에서 읽기

```kotlin
// Command: 원칙대로 루트를 통해 변경
val order = orderRepository.findById(orderId)
order.changeShippingInfo(newInfo)

// Query: 화면에 배송정보만 보여줄 때는 직접 조회 OK
val shippingInfo = queryDao.findShippingInfoByOrderId(orderId)
```
조회는 데이터 일관성을 깨뜨릴 위험이 없으니까 자유롭게 해도 괜찮다고 판단.

#### 4. 리포지터리 (Repository) — "DB 접근을 숨기는 창구"

도메인 코드에서 "이 주문을 저장해줘", "주문 #1234 가져와줘" 할 때 쓰는 인터페이스. 애그리거트 단위로 저장/조회.

```kotlin
interface OrderRepository {
    fun findById(id: Long): Order?
    fun save(order: Order)
}
```
내부적으로 MongoDB를 쓰든 Redis를 쓰든 도메인 로직은 신경 안 씀. 평소에 쓰는 Spring Data의 Repository 인터페이스가 바로 이 패턴.

#### 5. 도메인 서비스 (Domain Service) — "여러 애그리거트가 협력하는 로직"

"결제 금액 계산"을 생각. 주문 금액 + 배달비 - 쿠폰 할인 - 포인트 사용. 이건 Order 혼자 할 수 있는 일이 아니다. 쿠폰 애그리거트, 포인트 애그리거트, 배달비 정책 등 여러 놈이 협력해야 한다.

```kotlin
// 도메인 서비스: 특정 엔티티에 속하지 않는 비즈니스 로직
class PaymentCalculationService {
    fun calculate(order: Order, coupon: Coupon?, point: Point): Money {
        val orderAmount = order.totalAmount()
        val discount = coupon?.calculateDiscount(orderAmount) ?: Money.ZERO
        val pointUsage = point.available()
        return orderAmount - discount - pointUsage + DeliveryFeePolicy.calculate(orderAmount)
    }
}
```

이걸 Order 엔티티 안에 넣으면 Order가 Coupon, Point까지 알아야 하니까 책임이 너무 커진다. 그래서 별도 서비스로 뺀다.

## 2.6 인프라스트럭처 개요

기술적 구현은 전부 인프라스트럭처임 
응용, 표현, 도메인 각 계층은 "뭘 할지"만 정의하고, "어떻게 할지"는 인프라가 담당

- 도메인 계층 지원
  - Repository 구현체 (JPA, MongoDB, Redis)
  - 도메인 이벤트 발행 구현 (Kafka, RabbitMQ)
- 응용 계층 지원
  - @Transactional → 스프링의 트랜잭션 매니저가 인프라야
  - 이메일 발송 → SMTP 클라이언트
  - 외부 API 호출 → REST Client, gRPC Client
  - 메시지 큐 발행 → KafkaTemplate
- 표현 계층 지원
  - Spring MVC 자체 (DispatcherServlet, Jackson 직렬화)
  - 인증/인가 (Spring Security, JWT 파싱)
  - CORS 설정, 필터 체인
