## 1.1 도메인이란??

**소프트웨어로 해결하고자 하는 비즈니스의 영역**

- 배달 앱을 만든다면? 도메인은 '음식 주문', '결제', '배달 경로 최적화' 등

- 쇼핑몰을 만든다면? 도메인은 '상품 관리', '재고 관리', '회원 프로모션' 등

## 1.2 도메인 전문가와 개발자 간 지식 공유

> **"전문가의 현장 지식"**과 **"개발자의 논리적 
설계"**를 하나의 언어로 합치는 과정이 바로 DDD의 핵심

제대로 소통이 안 되면 결국 "엉뚱한 물건"을 만들게 된다.

따라서 **보편적 언어(Ubiquitous Language)를 써라** 기획자는 "주문 취소"라고 하는데, 개발자는 코드에 `deleteOrder()`라고 써놓으면 나중에 대화가 꼬임

- 약속: 팀원 모두(기획자, 디자이너, 개발자)가 같은 단어를 쓰기로 약속
- 코드에도 활용 : 이 단어는 회의실 벽에만 붙어있는 게 아니라, 실제 변수명과 클래스명에도 그대로 들어가야 함
- 번역 금지 : 머릿속에서 '비즈니스 언어'를 '개발 용어'로 번역하는 순간 오해가 생김


## 1.3 도메인 모델

> 인 모델은 **"우리 서비스가 어떻게 돌아가는지 코드로 그린 그림"**이라고 할 수 있음 이 모델이 잘 잡혀 있어야 서비스가 복잡해져도 코드가 엉키지 않고 비즈니스의 요구사항을 정확히 반영

## 1.4 도메인 모델 패턴

도메인 모델 패턴을 사용하면 비즈니스 로직이 DB나 UI가 아닌 **'도메인 객체'**에 집중됨. 이렇게 하면 요구사항이 변경되었을 때 DB 테이블을 뒤지는 것이 아니라, 해당 비즈니스 언어를 사용하는 코드를 찾아 수정하면 되므로 유지보수가 매우 명확해짐.

```java
// [도메인 계층] 주문 상태를 나타내는 Value Object
public enum OrderState {
    PREPARING, SHIPPED, DELIVERING, DELIVERY_COMPLETED, CANCELED;
}

// [도메인 계층] 비즈니스 로직을 품고 있는 도메인 엔티티
public class Order {
    private Long id;
    private OrderState state;
    private ShippingInfo shippingInfo;

    // 배송지 변경 로직: 비즈니스 규칙이 도메인 모델 내부에 있음
    public void changeShippingInfo(ShippingInfo newShippingInfo) {
        verifyNotYetShipped(); // 비즈니스 규칙 검증
        this.shippingInfo = newShippingInfo;
    }

    // 주문 취소 로직
    public void cancel() {
        verifyNotYetShipped();
        this.state = OrderState.CANCELED;
    }

    private void verifyNotYetShipped() {
        if (state != OrderState.PREPARING) {
            throw new IllegalStateException("이미 배송이 시작되어 변경/취소가 불가능함");
        }
    }
}
```

- 약속: OrderState의 PREPARING, SHIPPED 같은 명칭은 개발자가 임의로 정한 것이 아니라, 기획자와 "배송 준비 중", "출고됨" 상태를 정의하기로 합의한 약속임. 모든 팀원은 이 상태값들이 무엇을 의미하는지 동일하게 이해해야 함.
- 번역 금지: 주소를 변경하는 기능을 `datePostCode()` `dateAddress()` 같은 기술적인 필드 단위로 번역하지 마시오. 비즈니스에서 말하는 행위 그대로 `hangeShippingInfo()`는 메서드명을 사용하여 의도를 명확히 드러내야 함.
- 코드에도 활용: "배송이 시작되면 주소를 바꿀 수 없다"는 비즈니스 규칙은 기획서에만 있는 게 아님. 위 코드의 verifyNotYetShipped() 메서드처럼 실제 코드의 로직으로 직접 활용되어 데이터의 일관성을 강제함.


## 1.5 도메인 모델 추출

비즈니스 요구사항
- "주문은 최소 결제 금액 10,000원 이상이어야 함."
- "배달이 시작(START)된 이후에는 주문 내용을 수정할 수 없음."
- "배달 상태는 대기(WAITING), 시작(START), 완료(COMPLETED)로 구분함."

```kotlin
// [도메인 계층] 주문 상태를 나타내는 Value Object
enum class DeliveryStatus(val description: String) {
    WAITING("배달 대기 중"),
    START("배달 시작됨"),
    COMPLETED("배달 완료");
}

// [도메인 계층] 돈의 개념을 담은 Value Object
data class Money(val amount: Long) {
    init {
        require(amount >= 0) { "금액은 0원 이상이어야 함" }
    }
    operator fun plus(other: Money) = Money(this.amount + other.amount)
}

// [도메인 계층] 배달 주문 엔티티 (핵심 도메인 모델)
class DeliveryOrder(
    val id: Long,
    private var status: DeliveryStatus = DeliveryStatus.WAITING,
    private var totalPrice: Money
) {
    companion object {
        private const val MINIMUM_ORDER_PRICE = 10000L
        
        // 주문 생성 로직: 최소 금액 검증 규칙 포함
        fun createOrder(id: Long, price: Money): DeliveryOrder {
            require(price.amount >= MINIMUM_ORDER_PRICE) { "최소 주문 금액은 ${MINIMUM_ORDER_PRICE}원임" }
            return DeliveryOrder(id = id, totalPrice = price)
        }
    }

    // 배달 시작 로직
    fun startDelivery() {
        this.status = DeliveryStatus.START
    }

    // 비즈니스 로직: 배달 시작 후에는 가격 수정 불가 규칙
    fun updatePrice(newPrice: Money) {
        check(status == DeliveryStatus.WAITING) { "배달이 시작된 후에는 가격을 변경할 수 없음" }
        this.totalPrice = newPrice
    }
}
```

- 약속: DeliveryStatus의 각 항목과 MINIMUM_ORDER_PRICE 같은 상수는 기획 단계에서 정의된 비즈니스 용어와 수치를 그대로 사용하기로 팀원들과 약속한 결과물임. 이를 통해 기획서와 코드 사이의 간극을 좁힐 수 있음.
- 코드에도 활용: 단순히 데이터를 담는 DTO가 아니라, updatePrice 메서드 내부의 check 문처럼 비즈니스 규칙이 코드의 로직으로 직접 활용되어야 함. 객체 스스로가 자신의 상태를 검증하도록 설계하는 것이 도메인 모델 패턴의 정수임.
- 번역 금지: "배달 대기 중"이라는 상태를 코드에서 status = 0으로 번역하지 마시오. 코틀린의 Enum을 활용하여 DeliveryStatus.WAITING이라고 명시함으로써, 개발자가 코드를 읽을 때 별도의 해석 없이 비즈니스 의도를 바로 이해할 수 있게 해야 함.

## 1.6 엔티티와 밸류

- 엔티티 (Entity): 식별자가 지배하는 객체엔티티는 속성이 변하더라도 **'고유한 식별자'**를 통해 그 정체성을 유지하는 객체

```kotlin
class Member(
    val id: Long, // 고유 식별자 (정체성)
    var name: String, // 변경 가능한 속성
    var address: String
) {
    // id가 같으면 같은 객체로 판별하는 로직이 들어감
    override fun equals(other: Any?): Boolean {
        if (this === other) return true
        if (other !is Member) return false
        return id == other.id
    }
}
```

- 밸류 객체 (Value Object): 속성값이 곧 정체성인 객체밸류 객체는 식별자 없이 '담고 있는 값' 자체로 정체성을 나타내는 객체임.

```kotlin
data class Address(
    val city: String,
    val zipCode: String
) // 모든 속성이 같으면 같은 값으로 취급함 (불변)
```

## 1.7 도메인 용어와 유비쿼터스 언어

> 1. 도메인 용어 (Domain Terminology)
도메인 용어는 특정 비즈니스 영역에서 전문가들이 일상적으로 사용하는 전문 용어를 의미함. 예를 들어 증권 도메인이라면 '예수금', '미수금', '매수' 같은 단어들이 이에 해당함.

>2. 유비쿼터스 언어 (Ubiquitous Language)
유비쿼터스 언어는 도메인 전문가와 개발자, 그리고 모든 이해관계자가 공통으로 사용하기로 정의한 단어들의 집합임. "유비쿼터스"라는 말처럼, 이 언어는 회의실, 문서, 그리고 코드 어디에서나 똑같이 나타나야 함

유비쿼터스 언어 사용의 효과
- 용어의 모호성 제거: "사용자"라고 부를 때 그것이 '로그인 계정'인지, '구매 고객'인지, '배송 수신인'인지 명확히 구분할 수 있게 됨.
- 문서와 코드의 일치: 기획서에 적힌 단어가 코드에 그대로 등장하므로, 코드가 곧 살아있는 문서가 됨.
- 지식의 축적: 개발자가 비즈니스 로직을 깊이 이해하게 되어, 기술적인 관점뿐만 아니라 비즈니스 관점에서도 더 나은 제안을 할 수 있음.
