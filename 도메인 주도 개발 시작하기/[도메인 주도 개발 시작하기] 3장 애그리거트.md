## 3.1 애그리거트

애그리거트 설계 규칙 

1. 한 애그리거트에 한 루트 엔티티모든 애그리거트는 반드시 루트 엔티티 하나가 대장이 된다. 즉, 외부에서는 루트만 참조할 수 있다.

// ✅ 외부에서는 Order(루트)만 접근

// ❌ 내부 객체를 직접 조회하면 안 됨
orderLineRepository.findById(orderLineId)  // 이런 리포지터리 자체가 없어야 함
```

## 3.2 애그리거트 루트

애그리거트는 내부에서 일관되게 모두 변경되어야하고, 전체를 관리할 주체가 필요하다.

```kotlin
[Order 애그리거트]
┌──────────────────────────────┐
│  🏷 Order  ← 애그리거트 루트     │  ← 외부에서 접근 가능한 건 얘뿐
│    ├── OrderLine (VO)        │
│    ├── Orderer (VO)          │
│    └── ShippingInfo (VO)     │
└──────────────────────────────┘
```

루트가 하는 일 3가지

**1. 외부와의 유일한 창구
**
```kotlin
// ✅ 외부에서는 루트만 조회
val order = orderRepository.findById(orderId)

// ❌ 내부 객체를 직접 꺼내면 안 됨
val orderLine = orderLineRepository.findById(lineId)  // 이런 거 없어야 함
```

**2.내부 변경의 문지기 (비즈니스 규칙 검증)
**
```kotlin
class Order(
    private var status: OrderStatus,
    private var shippingInfo: ShippingInfo
) {
    fun changeShippingInfo(newInfo: ShippingInfo) {
        // 루트가 규칙을 검증한 뒤에만 내부 상태를 변경
        if (status == OrderStatus.DELIVERING)
            throw IllegalStateException("배달 중엔 변경 불가")
        this.shippingInfo = newInfo
    }
}
```
만약 외부에서 shippingInfo를 직접 바꿀 수 있으면? 배달 중인데 주소가 바뀌는 사고가 날 수 있다. 루트가 문지기 역할을 해서 일관성을 지키는 것

**3.애그리거트의 식별자 역할
**
애그리거트 전체를 대표하는 ID를 루트가 가진다. 외부에서 이 애그리거트를 참조할 때는 루트의 ID를 사용

```kotlin
class Payment(
    val id: Long,
    val orderId: Long  // Order 애그리거트의 루트 ID로 참조
)
```

내부 객체들의 라이프사이클을 관리 (Order가 삭제되면 OrderLine도 같이 삭제)하고, 비즈니스 규칙을 검증하고 일관성을 보장하는 주체가 애그리거트 루트이다.

### 3.2.1 도메인 규칙과 일관성

의미없는 `setter`를 제거하라.

```kotlin
// ❌ 공개 setter — 아무 규칙 없이 값만 바꿈
class Order {
    var shippingInfo: ShippingInfo = ...  // 아무나 order.shippingInfo = xxx 가능
    var status: OrderStatus = ...         // order.status = COMPLETED 이렇게 막 바꿈
}

// 외부에서 이렇게 쓰면 비즈니스 규칙이 무시됨
order.shippingInfo = newInfo  // 배달 중인데도 바꿔버릴 수 있음
order.status = OrderStatus.COMPLETED  // 결제도 안 했는데 완료?
```

```kotlin
// ✅ 의미 있는 메서드 — 비즈니스 의도를 담음
class Order(
    private var shippingInfo: ShippingInfo,  // private!
    private var status: OrderStatus
) {
    fun changeShippingInfo(newInfo: ShippingInfo) {  // "배송지 변경"이라는 의도
        if (status == OrderStatus.DELIVERING)
            throw IllegalStateException("배달 중엔 변경 불가")
        this.shippingInfo = newInfo
    }

    fun complete() {  // "주문 완료"라는 의도
        if (status != OrderStatus.DELIVERING)
            throw IllegalStateException("배달 중 상태에서만 완료 가능")
        this.status = OrderStatus.COMPLETED
    }
}
```

객체가 불변일 경우에는 새로운 객체로 교체하여야 한다.

```kotlin
// 사용할 때
val newInfo = ShippingInfo("서울시 강남구", "홍길동", "010-1234-5678")
order.changeShippingInfo(newInfo)  // 새 객체를 만들어서 넘김

// 일부만 바꾸고 싶으면 Kotlin copy() 활용
val updated = order.shippingInfo.copy(address = "서울시 서초구")
order.changeShippingInfo(updated)
```

`VO` 자체는 불변을 유지하면서도 애그리거트의 상태는 변경할 수 있다. 불변인 건 VO이고, 그 VO를 들고 있는 **루트 엔티티의 참조는 가변(var)**이다.

이렇게 하면 VO가 외부에서 몰래 변경되는 부작용(side effect)을 원천 차단할 수 있어서, 멀티스레드 환경에서도 안전해진다. 

### 3.2.2 애그리거트 루트의 기능 구현

애그리거트 루트는 다른 애그리거트 내부의 다른 객체를 조합하여 기능을 완성한다.

```kotlin
data class CartItem(
    val productId: Long,
    val productName: String,
    val price: Money,
    val quantity: Int
) {
	// 값 계산은 CartItem이 위임
    fun totalPrice(): Money = price * quantity
    fun isOutOfStock(stock: Int): Boolean = quantity > stock
}

class Cart(
    val id: Long,
    private val memberId: Long,
    private val items: MutableList<CartItem> = mutableListOf()
) {
    // 내부 CartItem들을 조합해서 총 금액
    fun totalAmount(): Money = items.sumOf { it.totalPrice() }

    // 내부 CartItem들을 조합해서 주문 가능 여부 판단
    fun canOrder(stockMap: Map<Long, Int>): Boolean {
        if (items.isEmpty()) return false
        return items.none { it.isOutOfStock(stockMap[it.productId] ?: 0) }
    }

    // CartItem + 비즈니스 규칙 조합해서 상품 추가
    fun addItem(newItem: CartItem) {
        if (items.size >= 50)
            throw IllegalStateException("장바구니는 최대 50개")

        val existing = items.find { it.productId == newItem.productId }
        if (existing != null) {
            // 같은 상품이면 수량 합산한 새 객체로 교체 (VO니까)
            items.remove(existing)
            items.add(existing.copy(quantity = existing.quantity + newItem.quantity))
        } else {
            items.add(newItem)
        }
    }
}
```

### 3.2.2 트랜잭션 범위

핵심 원칙은 하나다: 한 트랜잭션에서 한 애그리거트만 수정한다.

```kotlin
// ✅ 한 트랜잭션에서 Order만 수정
@Transactional
fun placeOrder(cmd: PlaceOrderCommand) {
    val order = Order.create(cmd)
    orderRepository.save(order)
    // 끝. 이 트랜잭션에서는 Order만 건드림

    eventPublisher.publish(OrderPlacedEvent(
        orderId = order.id,
        productId = cmd.productId,
        quantity = cmd.quantity,
        memberId = cmd.memberId,
        totalAmount = order.totalAmount()
    ))
}

// 각각 별도 트랜잭션에서 자기 애그리거트만 수정
@TransactionalEventListener
fun onOrderPlaced_decreaseStock(event: OrderPlacedEvent) {
    val product = productRepository.findById(event.productId)
    product.decreaseStock(event.quantity)  // Product만 수정
}

@TransactionalEventListener
fun onOrderPlaced_addPoint(event: OrderPlacedEvent) {
    val member = memberRepository.findById(event.memberId)
    member.addPoint(event.totalAmount * 0.01)  // Member만 수정
}
```

```
[트랜잭션 1] Order 생성 → 커밋 → 이벤트 발행
                                    ↓
[트랜잭션 2] Product 재고 차감 ─────────┤
[트랜잭션 3] Member 포인트 적립 ────────┘
```

이러면 포인트 적립이 실패해도 주문과 재고는 영향 안 받는다.

__그러나 최종 일관성의 경우에는 다를 수 있다.__
재고 차감이 실패하면?

- **즉시 일관성 vs 최종 일관성**
```
한 애그리거트 내부    → 즉시 일관성 (같은 트랜잭션, 무조건 맞아야 함)
애그리거트 간   → 최종 일관성 (잠깐은 안 맞아도 결국은 맞으면 됨)
```

따라서 최종 일관성을 위해서 재시도, 보상 트랜잭션, Outbox 패턴 (이벤트 유실 방지) 등을 활용해야 한다.

---

추가적으로 도메인 이벤트처리를 통해 다른 애그리거트를 동시에 처리할 수 있다.

도메인 이벤트를 처리하기 위해서 Spring에서는 2개의 어노테이션을 제공한다.

![](https://velog.velcdn.com/images/cksgodl/post/1f33f299-321f-48f7-b586-3ed2699c3450/image.png)


```
@Transactional 시작
│
├── Order.complete()  →  커밋!  ✅ 주문 완료 확정
│                           │
│                           ▼  (커밋 이후에 실행)
│                    @TransactionalEventListener
│                    ├── 카카오 알림톡 발송 (실패해도 주문은 OK)
│                    ├── 포인트 적립 (실패해도 주문은 OK)
│                    └── 통계 업데이트 (실패해도 주문은 OK)
```

```kotlin
// ② Application Service — 주문 완료 + 이벤트 발행
@Service
class OrderService(
    private val orderRepository: OrderRepository,
    private val eventPublisher: ApplicationEventPublisher
) {
    @Transactional
    fun completeOrder(orderId: Long) {
        val order = orderRepository.findById(orderId)
            ?: throw OrderNotFoundException(orderId)

        order.complete()  // Order만 수정, 여기서 커밋

        eventPublisher.publishEvent(OrderCompletedEvent(
            orderId = order.id,
            memberId = order.memberId,
            totalAmount = order.totalAmount(),
            memberEmail = order.ordererEmail
        ))
    }
}
```

```kotlin
// ③ 이벤트 핸들러들 — 커밋 후 실행
@Component
class OrderCompletedEventHandler(
    private val notificationSender: NotificationSender,
    private val memberRepository: MemberRepository,
    private val statisticsService: StatisticsService
) {
    // 알림 발송 — 커밋 후 실행, 실패해도 주문에 영향 없음
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    fun sendNotification(event: OrderCompletedEvent) {
        notificationSender.sendOrderComplete(
            email = event.memberEmail,
            orderId = event.orderId
        )
    }

    // 포인트 적립 — 별도 트랜잭션
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    fun addPoint(event: OrderCompletedEvent) {
        val member = memberRepository.findById(event.memberId)
        member.addPoint(event.totalAmount * 0.01)
    }

    // 통계 업데이트
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    fun updateStatistics(event: OrderCompletedEvent) {
        statisticsService.recordOrderComplete(
            orderId = event.orderId,
            amount = event.totalAmount
        )
    }
}
```

## 3.3 리포지토리와 애그리거트

핵심 원칙: 리포지터리는 애그리거트 단위로 존재한다.

```kotlin
// ✅ 애그리거트 루트 단위로 리포지터리 존재
interface OrderRepository {
    fun findById(id: Long): Order?
    fun save(order: Order)
    fun delete(order: Order)
}

// ❌ 내부 VO/엔티티에 대한 리포지터리는 없어야 함
interface OrderLineRepository { ... }     // 이런 거 만들면 안 됨
interface ShippingInfoRepository { ... }  // 이것도 안 됨
```

```kotlin
// Order를 저장하면 내부 OrderLine, ShippingInfo도 같이 저장
orderRepository.save(order)

// Order를 조회하면 내부 객체도 같이 딸려옴
val order = orderRepository.findById(orderId)
// order.orderLines → 이미 로딩됨
// order.shippingInfo → 이미 로딩됨

// Order를 삭제하면 내부 객체도 같이 삭제
orderRepository.delete(order)
```

- 애그리거트 = 저장의 단위
- 리포지터리 = 그 단위를 저장/조회하는 창구
- 애그리거트 루트 1개 = 리포지터리 1개
- 내부 객체는 루트와 함께 저장/조회/삭제

레포지토리 구현의 경우는 사용하는 데이터베이스에 따라 다르다.

```kotlin
@Entity
class Order(
    @Id val id: Long,

    // cascade: Order 저장/삭제 시 OrderLine도 함께
    // orphanRemoval: Order에서 빠진 OrderLine은 자동 삭제
    @OneToMany(cascade = [CascadeType.ALL], orphanRemoval = true)
    private val orderLines: MutableList<OrderLine> = mutableListOf(),

    @Embedded
    private var shippingInfo: ShippingInfo
)

// save 한 번이면 Order + OrderLine + ShippingInfo 전부 저장
orderRepository.save(order)
```

## 3.4 ID를 통한 애그리거트 참조

```kotlin
class Order(
    val id: OrderId,
    val customerId: CustomerId,  // ID만 보관
    val items: List<OrderItem>
)

한 애그리거트에서 다른 애그리거트를 참조할 수 있다.

### __JPA의 @ManyToOne, @OneToOne과 ORM__
> ORM이란?
Object-Relational Mapping — 객체 세계와 관계형 DB 세계 사이의 패러다임 불일치를 자동으로 해소해주는 기술
```
[ 객체 세계 ]          [ 관계형 DB 세계 ]
  상속                  없음 (조인으로 흉내)
  참조 (레퍼런스)        외래키 (FK)
  다형성                없음
  그래프 탐색           SQL 조인
```
JPA는 이 불일치를 어노테이션 매핑으로 처리하고, Hibernate가 실제 SQL을 생성


앞서 얘기한 ID 참조 원칙과 JPA 연관관계 매핑은 충돌할 수 있다.

```kotlin
// JPA 스타일 — 객체 참조
@ManyToOne
@JoinColumn(name = "customer_id")
val customer: Customer  // 바로 접근 가능하기에 애그리거트 경계 침범

// DDD 스타일 — ID 참조
@Column(name = "customer_id")
val customerId: Long    // 경계 명확
```

보통 두 가지 전략이 있다.

```
같은 애그리거트 내부        → @ManyToOne / @OneToOne 사용 (Order ↔ OrderItem)
다른 애그리거트 참조        → ID만 저장 (Order → customerId: Long)
```

### Eager vs Lazy Loading

핵심 개념: 프록시
Lazy 로딩을 이해하려면 먼저 프록시를 알아야 한다.
```kotlin
val order = orderRepository.findById(1L) // 이 시점엔 customer를 아직 안 가져옴

// 내부적으로 customer는 이런 프록시 객체로 대체됨
class Customer$HibernateProxy : Customer() {
    private var _loaded = false
    private var _real: Customer? = null

    override fun getName(): String {
        if (!_loaded) {
            _real = // 이 시점에 SELECT * FROM customer WHERE id = ? 실행
            _loaded = true
        }
        return _real!!.getName()
    }
}
```
- Lazy = 실제 필드에 접근하는 순간 SQL을 날리는 가짜 객체를 껍데기로 넣어두는 것
- Eager — 즉시 로딩

```kotlin
@ManyToOne(fetch = FetchType.EAGER) // JPA 기본값
val customer: Customer
```

```sql
-- orderRepository.findById(1L) 호출 시
SELECT o.*, c.*
FROM orders o
LEFT OUTER JOIN customer c ON o.customer_id = c.id
WHERE o.id = 1
```
Order를 꺼내는 순간 Customer를 무조건 JOIN해서 같이 가져옴

```kotlin
@ManyToOne(fetch = FetchType.LAZY) // 권장
val customer: Customer
```

### Lazy로딩의 N+1 문제

1번 조회해야 할 것을, 연관 데이터 때문에 N번 추가로 더 조회하는 문제

도서관에서 책 10권의 저자 정보를 알고 싶다.


```
// N+1 방식 (나쁜 방식)
"책 목록 전부 가져와" → 책 10권 수령 (1번)
"1번 책 저자 정보 가져와" → 수령 (+1)
"2번 책 저자 정보 가져와" → 수령 (+1)
... 10번 반복

→ 총 11번 도서관 왕복

// 올바른 방식

"책 10권이랑 저자 정보 같이 가져와" → 한 번에 수령

→ 총 1번 도서관 왕복
```

```kotlin
// Order 10개를 조회하고, 각 주문의 고객 이름을 출력하는 상황
val orders = orderRepository.findAll()

orders.forEach {

}
실제로 날아가는 SQL:
sql-- 1번: 주문 목록 조회
SELECT * FROM orders;

-- +1번: order_id=1의 customer 조회
SELECT * FROM customer WHERE id = 1;

-- +1번: order_id=2의 customer 조회
SELECT * FROM customer WHERE id = 2;

-- +1번: order_id=3의 customer 조회
SELECT * FROM customer WHERE id = 3;

-- ... 10개면 총 11번, 100개면 101번, 1000개면 1001번
```

#### 왜 발생하는가

JPA가 `orders`를 가져올 때, `customer`는 **프록시(빈 껍데기)** 로 채워지기 때문
```kotlin
Order(id=1, customer=Proxy)   ← 진짜 Customer 아님
Order(id=2, customer=Proxy)
Order(id=3, customer=Proxy)
```

그 다음 루프를 돌면서 customer.name에 처음 접근하는 순간 각 프록시가 DB에 쿼리를 날린다. JPA 입장에서는 "요청받았으니까 가져오는 것" — 최적화를 못 한 채 루프마다 각자 쿼리를 실행

- 해결: fetch join

```kotlin
@Query("SELECT o FROM Order o JOIN FETCH o.customer")
fun findAllWithCustomer(): List<Order>
```

```sql
-- 단 1번의 쿼리로 해결
SELECT o.*, c.*
FROM orders o
JOIN customer c ON o.customer_id = c.id;
```

처음부터 Customer를 **JOIN해서 함께 로딩**하기 때문에 루프를 돌아도 추가 쿼리가 없음

#### 왜 위험한가

```
주문 10개    → 쿼리 11번   (감내 가능)
주문 1,000개 → 쿼리 1,001번 (느려짐)
주문 10,000개 → 쿼리 10,001번 (💥 서비스 장애)
```

팀처럼 **초당 수백 건**을 처리하는 광고 서빙 시스템에서는 N+1이 터지는 순간 DB 커넥션 고갈로 바로 장애로 이어질 수 있음.

### DDD 조인 (아이디 조인)의 활용

하위 도메인마다 서로 다른 DBMS활용 가능. 따라서 ID를 통해서 다른 애그리거트를 참조하는 것을 권장

## 3.5 애그리거트 간 집한 연관

```
애그리거트 내부  → 객체 참조 (@OneToMany 등) OK
애그리거트 간    → ID 참조만
```

- 1-N 연관
애그리거트 내부 1-N (객체 참조 OK)
같은 애그리거트 루트가 관리하는 자식들
```kotlin
@Entity
class Order(
    @Id @GeneratedValue
    val id: Long = 0,

    @OneToMany(
        cascade = [CascadeType.ALL],
        orphanRemoval = true  // Order에서 제거하면 DB에서도 삭제
    )
    @JoinColumn(name = "order_id")
    val items: MutableList<OrderItem> = mutableListOf()
) {
    // 불변식을 루트가 직접 보호
    fun addItem(productId: Long, quantity: Int, price: Int) {
        check(quantity > 0) { "수량은 0보다 커야 합니다" }
        items.add(OrderItem(productId, quantity, price))
    }

    fun totalAmount(): Int = items.sumOf { it.price * it.quantity }
}

@Entity
class OrderItem(  // 애그리거트 루트 없이 단독 존재 불가
    @Id @GeneratedValue
    val id: Long = 0,
    val productId: Long,
    val quantity: Int,
    val price: Int
)
```

- 애그리거트 간 1-N (ID 참조 필수)

```kotlin
// ❌ 잘못된 방식 — 다른 애그리거트를 컬렉션으로 직접 참조
@Entity
class Customer(
    val id: Long,
    
    @OneToMany  // Order는 다른 애그리거트인데 직접 참조
    val orders: List<Order>  // 트랜잭션 경계 붕괴, 로딩 폭탄
)
```

```kotlin
// ✅ 올바른 방식 — Order가 customerId를 들고 있음
@Entity
class Order(
    @Id @GeneratedValue
    val id: Long = 0,

    @Column(name = "customer_id")
    val customerId: Long,  // ID만 참조

    @OneToMany(cascade = [CascadeType.ALL], orphanRemoval = true)
    @JoinColumn(name = "order_id")

)

@Entity
class Customer(

    val id: Long = 0,

    // orders 컬렉션 없음 — 필요하면 Repository로 조회
)
```

- M-N 연관

#### 문제: M-N은 항상 중간 테이블이 필요

```
Product ─────────── Category
  1개 상품은 여러 카테고리
  1개 카테고리엔 여러 상품
```
JPA의 @ManyToMany는 쓰면 안됨 -> 이유:


```kotlin
// ❌ @ManyToMany — 실무에서 금지
@Entity
class Product(
    @ManyToMany
    @JoinTable(name = "product_category")
    val categories: MutableList<Category>
)
```

- 중간 테이블에 추가 컬럼(등록일, 등록자 등)을 넣을 수 없음
- 양방향 동기화 관리가 복잡해짐
- N+1 폭탄

해결: 중간 테이블을 별도 엔티티로 승격

```kotlin
// 중간 엔티티로 M-N을 두 개의 1-N으로 분해
@Entity
class ProductCategory(
    @Id @GeneratedValue
    val id: Long = 0,

    @Column(name = "product_id")
    val productId: Long,      // ID 참조

    @Column(name = "category_id")
    val categoryId: Long,     // ID 참조

    val assignedAt: LocalDateTime = LocalDateTime.now(),
    val assignedBy: String    // 중간 테이블에 부가 정보 추가 가능
)
```

```kotlin
@Entity
class Product(
    @Id @GeneratedValue
    val id: Long = 0,
    val name: String,

    @OneToMany(cascade = [CascadeType.ALL], orphanRemoval = true)
    @JoinColumn(name = "product_id")
    val productCategories: MutableList<ProductCategory> = mutableListOf()
) {
    fun assignCategory(categoryId: Long, assignedBy: String) {
        val alreadyAssigned = productCategories.any { it.categoryId == categoryId }
        check(!alreadyAssigned) { "이미 등록된 카테고리입니다" }
        productCategories.add(ProductCategory(productId = id, categoryId = categoryId, assignedBy = assignedBy))
    }

    fun removeCategory(categoryId: Long) {
        productCategories.removeIf { it.categoryId == categoryId }
    }
}
```

- 전체 구조 한눈에 보기
```
[ 애그리거트 내부 1-N ]               [ 애그리거트 간 1-N ]
  Order ──@OneToMany──> OrderItem   Order ──customerId──> Customer
  (객체 참조 OK,                      (ID 참조만,
   같은 트랜잭션,                       별도 트랜잭션,
   cascade ALL)                      Repository로 조회)

[ 애그리거트 간 M-N ]
  Product ──@OneToMany──> ProductCategory <──(categoryId)── Category
  (중간 엔티티로 승격,
   부가 정보 추가 가능,
   @ManyToMany 사용 금지)
```

## 3.6 애그리거트를 팩토리로 사용하기

왜 팩토리가 필요한가?
객체 생성이 복잡해지면 생성 로직 자체가 도메인 지식이 된다.

```kotlin
// ❌ 애플리케이션 서비스에서 직접 생성 — 도메인 지식이 밖으로 샘
class OrderApplicationService {
    fun createReview(customerId: Long, productId: Long, content: String) {
        // 리뷰 작성 가능 여부 판단 로직이 서비스에 노출됨
        val order = orderRepository.findCompletedByCustomerAndProduct(customerId, productId)
            ?: throw IllegalStateException("구매 이력이 없습니다")
        
        if (order.reviewedAt != null) throw IllegalStateException("이미 리뷰를 작성했습니다")
        if (order.completedAt.plusDays(30) < LocalDateTime.now()) throw IllegalStateException("리뷰 작성 기간이 지났습니다")

        // 이 조건들은 Order 도메인이 알아야 할 지식인데 서비스가 알고 있음
        val review = Review(customerId, productId, content)
        reviewRepository.save(review)
    }
}
```

"어떤 조건에서 Review를 만들 수 있는가" — 이건 Order가 알아야 할 도메인 지식이기 때문에 팩토리 패턴으로 추출

관련 있는 애그리거트가 다른 애그리거트의 생성 책임을 가지게하는 방식.

```kotlin
@Entity
class Order(
    @Id @GeneratedValue
    val id: Long = 0,

    val customerId: Long,
    val productId: Long,
    val status: OrderStatus,
    val completedAt: LocalDateTime?,
    val reviewedAt: LocalDateTime?
) {
    // Order가 Review 생성의 팩토리 역할
    fun createReview(content: String): Review {
        // 생성 조건(도메인 규칙)을 Order가 직접 보호
        check(status == OrderStatus.COMPLETED) { "완료된 주문만 리뷰 작성 가능합니다" }
        check(reviewedAt == null) { "이미 리뷰를 작성했습니다" }
        check(completedAt!!.plusDays(30) >= LocalDateTime.now()) {
            "리뷰 작성 기간(30일)이 지났습니다"
        }

        return Review(
            customerId = this.customerId,
            productId = this.productId,
            orderId = this.id,
            content = content
        )
    }
}
```

애플리케이션 서비스는 단순해짐

```kotlin
class OrderApplicationService(
    private val orderRepository: OrderRepository,
    private val reviewRepository: ReviewRepository
) {
    @Transactional
    fun createReview(orderId: Long, content: String) {
        val order = orderRepository.findById(orderId)

        val review = order.createReview(content) // 조건 판단은 Order가 담당

        reviewRepository.save(review)
    }
}
```

### 다양한 패턴

- 패턴 1: 상태 기반 생성

현재 애그리거트의 상태를 바탕으로 새 애그리거트를 만드는 패턴

```kotlin
@Entity
class Cart(
    @Id @GeneratedValue
    val id: Long = 0,

    val customerId: Long,

    @OneToMany(cascade = [CascadeType.ALL])
    val items: MutableList<CartItem> = mutableListOf()
) {
    // Cart가 Order를 생성하는 팩토리
    fun checkout(shippingAddress: Address): Order {
        check(items.isNotEmpty()) { "장바구니가 비어있습니다" }
        check(items.size <= 50) { "한 번에 최대 50개까지 주문 가능합니다" }

        return Order(
            customerId = this.customerId,
            items = items.map { OrderItem(it.productId, it.quantity, it.price) },
            shippingAddress = shippingAddress,
            status = OrderStatus.PENDING
        )
        // Cart 자체는 건드리지 않음 — 서비스에서 별도로 처리
    }
}
```

```kotlin
@Transactional
fun checkout(cartId: Long, address: Address) {
    val cart = cartRepository.findById(cartId)

    val order = cart.checkout(address) // Cart → Order 생성

    orderRepository.save(order)
    cartRepository.delete(cart)        // cart 정리는 서비스가 담당
}
```

- 패턴 2: 파생 애그리거트 생성
원본 애그리거트에서 파생되는 새로운 애그리거트를 만드는 패턴


```kotlin
@Entity
class Order(
    val id: Long,
    val customerId: Long,
    val items: List<OrderItem>,
    val status: OrderStatus
) {
    // Order에서 반품 요청 생성
    fun requestReturn(reason: String, itemIds: List<Long>): ReturnRequest {
        check(status == OrderStatus.DELIVERED) { "배송 완료 후에만 반품 가능합니다" }
        check(items.any { it.id in itemIds }) { "주문에 없는 상품입니다" }

        val returnItems = items.filter { it.id in itemIds }

        return ReturnRequest(
            orderId = this.id,
            customerId = this.customerId,
            items = returnItems.map { ReturnItem(it.productId, it.quantity) },
            reason = reason
        )
    }

    // Order에서 재주문 생성
    fun reorder(): Order {
        check(status == OrderStatus.COMPLETED) { "완료된 주문만 재주문 가능합니다" }

        return Order(
            customerId = this.customerId,
            items = this.items.map { it.copy() }, // 기존 항목 복사
            status = OrderStatus.PENDING
        )
    }
}
```

### 팩토리 메서드 vs 도메인 서비스 팩토리

생성 로직이 복잡해서 여러 애그리거트가 필요하다면 도메인 서비스로 분리한다.

```kotlin
// 단일 애그리거트로 생성 가능 → 팩토리 메서드
val review = order.createReview(content)

// 여러 애그리거트 정보가 필요 → 도메인 서비스
class CreativeFactory(
    private val campaignRepository: CampaignRepository,
    private val productRepository: ProductRepository
) {
    fun create(campaignId: Long, productId: Long, assets: List<CreativeAsset>): Creative {
        val campaign = campaignRepository.findById(campaignId)
        val product = productRepository.findById(productId)

        // 두 애그리거트를 모두 참조해야 검증 가능한 규칙
        check(product.advertiserId == campaign.advertiserId) {
            "캠페인과 상품의 광고주가 일치하지 않습니다"
        }
        check(product.isAvailableForAd()) { "광고 불가 상품입니다" }

        return campaign.createCreative(productId, assets, CreativeFormat.DISPLAY)
    }
}
```

판단 기준:
```
생성 조건이 하나의 애그리거트 상태로 판단 가능   → 팩토리 메서드
생성에 여러 애그리거트 정보가 필요              → 도메인 서비스 팩토리
```

      생성 로직이 애그리거트에 있으면 도메인 모델이 스스로를 보호하는 것
