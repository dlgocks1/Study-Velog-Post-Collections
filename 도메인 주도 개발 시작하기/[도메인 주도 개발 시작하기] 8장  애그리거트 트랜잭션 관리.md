## 8.1 애그리거트와 트랜잭션

한 애그리거트를 여러 사용자가 동시에 조회·수정하는 경우 발생
트랜잭션을 사용해도 애그리거트의 일관성이 깨질 수 있음

![](https://velog.velcdn.com/images/cksgodl/post/b33fc445-451a-4d76-91f1-285cd829ea9a/image.png)


- 트랜잭션은 각 변경 작업의 원자성만 보장
- 운영자와 고객의 변경 작업 사이 순서까지는 보장하지 않음
- 두 사용자가 모두 자신이 본 데이터 기준으로 작업했지만 결과는 어긋남


## 8.2 선점 잠금

왜 선점 잠금이 필요한가 — 두 명이 동시에 주문을 처리할 때


![](https://velog.velcdn.com/images/cksgodl/post/6922b121-3b29-45a3-b6a2-9a2f4f974c2d/image.png)

```kotlin
interface OrderRepository : Repository<Order, OrderNo> {
    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("select o from Order o where o.id = :id")
    fun findByIdForUpdate(@Param("id") id: OrderNo): Order?
}
```

LockModeType 종류 (책 기준)

- PESSIMISTIC_WRITE: 쓰기 잠금 — 가장 일반적으로 사용. 다른 트랜잭션의 읽기/쓰기 모두 차단
- PESSIMISTIC_READ: 읽기 잠금 — 공유 잠금. 데이터를 일관되게 읽고 싶지만 변경은 막고 싶지 않을 때
- PESSIMISTIC_FORCE_INCREMENT: 선점 잠금 + 버전 증가 (낙관적 잠금과 결합)

> 선점 잠금은 트랜잭션 종료 시점에 자동 해제 따라서 @Transactional 안에서 호출해야 의미가 있음.

데드락이 발생할 수 있음 -> 책에서는 javax.persistence.lock.timeout 힌트로 최대 대기 시간을 지정하라고 권함

## 8.3 비선점 잠금

선점 잠금이 "미리 막아둔다" 였다면, 비선점 잠금은 "일단 진행하고, 충돌이 생기면 그때 거부한다" 는 접근. DB 락을 걸지 않고 버전 번호로 충돌을 감지

![](https://velog.velcdn.com/images/cksgodl/post/462c16d4-2e5e-4269-a047-ee3e5da04418/image.png)

엔티티에 @Version 필드 하나를 추가, JPA가 자동으로 버전 증가와 검증 SQL을 처리

```kotlin
@Entity
@Table(name = "purchase_order")
class Order(
    @EmbeddedId
    val number: OrderNo,

    @Embedded
    var shippingInfo: ShippingInfo,

    @Enumerated(EnumType.STRING)
    var state: OrderState,

    @Version
    val version: Long = 0  // JPA가 자동 관리
) {
    fun changeShippingInfo(newInfo: ShippingInfo) {
        this.shippingInfo = newInfo
    }
}
```

### 비선점 잠금은 충돌 시 예외 처리가 필수

책에서는 응용 서비스 또는 표현 영역에서 `OptimisticLockingFailureException(Spring)` 또는 `OptimisticLockException(JPA)`을 잡아 사용자에게 재시도 처리 유도

### 강제 버전 증가 — PESSIMISTIC_FORCE_INCREMENT

루트 애그리거트가 직접 수정되지 않더라도 하위 엔티티가 바뀌면 충돌로 간주해야 하는 경우가 있다.


![](https://velog.velcdn.com/images/cksgodl/post/8a40fea4-5bcf-4fcb-acbc-913d836f1988/image.png)

이때 사용하는 LockModeType이 OPTIMISTIC_FORCE_INCREMENT

```kotlin
@Lock(LockModeType.OPTIMISTIC_FORCE_INCREMENT)
@Query("select o from Order o where o.id = :id")
fun findByIdForUpdate(@Param("id") id: OrderNo): Order?
```

이렇게 하면 루트의 다른 필드가 변경되지 않았어도, 트랜잭션 종료 시 무조건 version을 1 증가시킨다. 즉, "이 애그리거트 상태가 어떻게든 변경되었음"을 다른 트랜잭션에게 알릴 수 있다.

## 8.4 오프라인 선점 잠금

지금까지 본 선점/비선점 잠금은 모두 하나의 트랜잭션 안에서만 동작. 하지만 실제 업무는 "수정 화면을 열고 → 데이터를 변경한 뒤 → 저장 버튼을 누른다" 처럼 여러 HTTP 요청에 걸쳐 일어난다.

오프라인 선점 잠금은 이 "여러 트랜잭션, 여러 요청" 범위에서 동작하는 잠금이다. 
(위키 문서 편집을 예시) -> A가 편집을 시작하면, B는 A가 끝낼 때까지 편집 화면조차 열 수 없도록 차단

### 사용자가 직접 LockManager 선언

![](https://velog.velcdn.com/images/cksgodl/post/db3944aa-021f-4ce6-8ff6-657e852cd37d/image.png)

책에서는 오프라인 선점 잠금을 LockManager 라는 인터페이스로 추상화하여 진행

```kotlin
interface LockManager {
    fun tryLock(type: String, id: String): LockId
    fun checkLock(lockId: LockId)
    fun releaseLock(lockId: LockId)
    fun extendLockExpiration(lockId: LockId, inc: Long)
}

data class LockId(val value: String)
```

![](https://velog.velcdn.com/images/cksgodl/post/7d851778-042e-4ddc-8968-ba5c4c418b51/image.png)

오프라인 선점 잠금에는 반드시 만료 시간이 필요. 사용자가 편집 화면을 열어둔 채 브라우저를 닫으면, 잠금이 영원히 남아 다른 사용자가 영영 편집하지 못하는 상황이 발생.


### DB 기반 LockManager 핵심 구조

```sql
CREATE TABLE locks (
    type        VARCHAR(255) NOT NULL,
    id          VARCHAR(255) NOT NULL,
    lockid      VARCHAR(255) NOT NULL,
    expiration_time DATETIME,
    PRIMARY KEY (type, id),
    UNIQUE KEY uk_lockid (lockid)
);
```

(type, id) 가 PK이므로 같은 대상에 대해 동시에 두 개의 잠금이 존재할 수 없습니다 — INSERT 시 PK 충돌이 발생하면 잠금 획득 실패.

```kotlin
// tryLock의 핵심 로직 (의사 코드)
fun tryLock(type: String, id: String): LockId {
    checkAlreadyLocked(type, id)  // 기존 잠금이 있고 미만료라면 예외
    val lockId = LockId(UUID.randomUUID().toString())
    locking(type, id, lockId)     // INSERT (실패 시 LockException)
    val expirationTime = LocalDateTime.now().plusMinutes(lockTimeout)
    setExpirationTime(lockId, expirationTime)
    return lockId
}
```
