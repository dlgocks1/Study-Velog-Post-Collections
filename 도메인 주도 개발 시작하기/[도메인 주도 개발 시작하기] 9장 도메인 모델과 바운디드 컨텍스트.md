## 9.1 도메인 모델과 경계

모델은 특정한 컨텍스트 하에서 완전한 의미를 같는다. 같은 제품이라도 카탈로그 컨텍스트와 재고 컨텍스트에서의 의미는 서로 다르다.

![](https://velog.velcdn.com/images/cksgodl/post/4227b3f6-1636-48e3-a275-dbfe7fedc8f2/image.png)

이러한 경계를 갖는 컨텍스트를 바운디드 컨텍스트라고 부른다.

## 9.2 바운디드 컨텍스트

바운디드 컨텍스트는 단순한 도메인 모델의 경계가 아니라, 사용자에게 기능을 제공하기 위한 모든 구성요소를 포함하는 구현체 (코드, 개념 포함)

![](https://velog.velcdn.com/images/cksgodl/post/9dcf5acf-aab2-4840-a87b-c64c95c87b46/image.png)

#### 각 BC마다 독립적으로 작동

각 BC마다 다른 아키텍처 선택 가능

- 핵심 도메인 → DDD + 헥사고날
- 단순 조회 위주 → CQRS 또는 트랜잭션 스크립트
- 데이터 변환만 → 단순 ETL



#### 적정 크기를 잡는 기준

- 너무 크다 → 다시 거대 단일 모델로 회귀
- 너무 작다 → BC 간 통신·정합성 비용 폭증
- 실무적 기준 → 한 팀이 온전히 책임질 수 있는 단위 (Two-Pizza Team)

### 핵심 정리

> BC = 모델 경계 + 구현 경계

도메인 전문가의 언어가 일관되게 통하는 영역 = 그 BC의 범위


## 9.3 바운디드 컨텍스트 구현

사용자에게 기능을 제공하기 위한 모든 구성요소를 포함하는 구현체

![](https://velog.velcdn.com/images/cksgodl/post/23f197bb-4058-459c-9d1f-5fb3f3fdee5d/image.png)

#### BC가 포함하는 것

- 표현 영역 (Controller, View)

## 9.4 바운디드 컨텍스트 간 통합

책 예시 추천 BC와 카탈로그 BC간 통합

![](https://velog.velcdn.com/images/cksgodl/post/6443056f-27cd-46d1-9a58-07a2cf55fc15/image.png)

위 다이어그램의 핵심은 RecSystemClient가 `ACL(안티코럽션 계층)` 역할을 한다는 점이다. 

추천 BC가 보내는 단순 모델(`{productId, score}`)이 카탈로그 도메인으로 그대로 흘러들지 못하게 막고, `List<Product>`로 번역해서 넘긴다.
  
도메인은 `ProductRecommendationService` 인터페이스에만 의존하므로, 외부 시스템이 바뀌어도 `ACL` 구현체만 고치면 된다.
  
  
#### ACL(Anti-Corruption Layer, 안티코럽션 계층)

> ACL은 내 도메인 모델을 외부 시스템의 모델로부터 보호하기 위한 격리 계층이다. 

외부 시스템(레거시 시스템, 다른 팀의 BC, 서드파티 API)의 모델과 내 도메인 모델 사이에 번역기를 끼워서, 외부 개념이 내 도메인 언어로 직접 침투하지 못하게 막는다. 

위의 구조는 헥사고날 아키텍처(`Hexagonal Architecture`, = 포트 앤 어댑터 Ports & Adapters) 의 전형적인 모습이다. 

![](https://velog.velcdn.com/images/cksgodl/post/5a27f60a-de71-4ebb-b530-b8c01daa63ff/image.png)

> 마이크로서비스에서의 ACL: 모놀리스를 분해하거나 레거시를 점진적으로 교체할 때 ACL이 특히 자주 등장한다. 
> 새 서비스 앞에 ACL을 두고 레거시 모델을 번역하면, 레거시를 한 번에 걷어내지 않고도 새 도메인을 깨끗하게 시작할 수 있다.
> 경우 ACL이 클래스 수준을 넘어 별도 서비스/모듈로 커지기도 한다.

![](https://velog.velcdn.com/images/cksgodl/post/022d8ed2-2f13-41fa-ac09-fedaf98edcdb/image.png)

- Shared Kernel (공유 커널): 두 팀이 모델의 일부를 공유한다. 결합도 최고. 한쪽이 공유 부분을 바꾸면 반드시 상대와 합의해야 한다. 긴밀한 두 팀에만 적합.
- Conformist (준수자): 하류가 상류 모델을 그대로 따른다. 번역 안 함, 방어 안 함. 비용 최저지만 상류에 완전 종속. 상류가 충분히 안정적이고 그 모델이 쓸 만할 때.
- OHS + PL (공개 호스트 서비스 + 공표된 언어): 상류가 잘 정의된 공개 계약(안정된 API + 중립적 스키마)을 제공. 하류는 그 계약만 믿으면 된다. 상류가 다수 하류를 상대할 때 효율적.
- ACL (안티코럽션 계층): 하류가 번역기를 직접 만들어 상류 모델을 막는다. 결합도 최저·방어 최대, 대신 번역 코드 유지 비용을 하류가 부담. 상류를 못 믿거나 모델이 더러울 때.


#### 메시지 큐를 활용한 비동기 통합

- REST가 동기·요청응답(pull) 방식
- 메시지 큐(Kafka, RabbitMQ 등)는 비동기·이벤트 기반(push) 방식

상류 BC가 도메인 이벤트를 발행하면, 하류 BC가 구독해서 자기 모델을 갱신한다. 두 컨텍스트가 직접 호출로 묶이지 않고 메시지 브로커를 사이에 두므로 시간적 결합(temporal coupling)이 사라진다.

![](https://velog.velcdn.com/images/cksgodl/post/1a407a2a-22a4-48e5-8c17-67ac90b764c6/image.png)

## 9.6 컨텍스트 맵

> 컨텍스트 맵 (Context Map)이란
>
> BC 간 관계들을 하나의 그림으로 모아 시스템 전체를 조망하는 지도다. 
> 개별 BC를 따로 보면 각자의 모델은 깨끗해 보이지만, 시스템이 커지면 "이 BC가 어떤 BC에 의존하고, 어디가 상류고, 어디에 방어막(ACL)을 쳐야 하는지"가 흐려진다. 

![](https://velog.velcdn.com/images/cksgodl/post/d182f7bb-120e-43e6-86ed-23178af2766e/image.png)

>라벨이 방어 지점을 알려준다. ACL이 붙은 엣지(추천 → 카탈로그)는 "여기서 외부 모델 오염을 막고 있다"는 뜻이고, Conformist가 붙은 엣지(카탈로그 → 주문)는 "여기선 주문 BC가 카탈로그 모델을 그대로 따른다 = 방어 안 함, 카탈로그 변경에 취약"이라는 경고로 읽힌다.


이러한 컨텍스트 맵은 조직 구조를 비춘다.

어떤 패턴이 붙느냐는 팀 관계를 반영한다(콘웨이의 법칙). 비협조적 상류 팀이 있으면 그 엣지엔 ACL이 붙기 마련이고, 맵을 보면 "어느 팀과의 협상이 시급한지"가 드러난다.
