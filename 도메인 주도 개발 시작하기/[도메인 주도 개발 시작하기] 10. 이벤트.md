## 10.1 시스템 간 강결합 문제

예시 핵심 사례 :  주문 취소 → 환불 같은, 한 도메인 기능이 외부 시스템을 호출해야 하는 상황

> 주문을 취소하면 결제(환불) 처리도 함께 일어나야 한다. 보통 이걸 주문 도메인 로직 안에서 외부 환불 서비스를 직접 호출하는 식으로 구현한다. 그런데 환불은 외부 시스템(PG사 등)이 담당하므로, 이 호출이 주문 도메인 코드 안으로 들어오는 순간 여러 문제가 한꺼번에 터진다.

- 설계상 결합 (외부 모델 침투): 주문 도메인 로직 안에 환불 서비스가 직접 등장
- 트랜잭션 범위 문제: 환불 처리를 어디까지 한 트랜잭션으로 묶을지 모호해진다.
- 성능 (응답 지연): 외부 환불 호출이 느리면 주문 취소 전체가 그만큼 느려진다. 외부 시스템 응답 시간이 내 기능의 응답 시간을 좌우한다.
- 장애 전파: 환불 시스템에 장애가 나거나 응답이 없으면, 그 예외가 주문 취소 로직까지 전파된다. 외부 장애가 내 기능 실패로 직결된다.

## 10.2 이벤트 개요

**본 강결합을 푸는 핵심 개념이 이벤트(Event) 다.**

### 이벤트 관련 구성 요소
이벤트가 동작하려면 네 가지 요소가 맞물린다.

- 이벤트 생성 주체 (도메인 객체): 엔티티, 밸류 등 도메인 객체가 도메인 로직을 실행한 뒤 이벤트를 생성한다. "주문 취소" 로직이 끝나면 OrderCanceled를 만든다.
- 이벤트 디스패처 (Publisher): 생성된 이벤트를 받아 핸들러에 전달하는 중개자. 생성 주체와 핸들러를 분리해준다.
- 이벤트 핸들러 (Subscriber/Handler): 이벤트에 반응해 후속 작업을 수행한다. OrderCanceled를 받아 환불을 처리하는 쪽.
- 이벤트 (Event 객체): 발생한 사실을 담은 데이터. 무슨 일이, 언제, 어떤 대상에 일어났는지를 표현한다.


![](https://velog.velcdn.com/images/cksgodl/post/a8ec9380-3abc-48c5-bec8-f0390ed77efe/image.png)

### 실 구현은?

- Spring에서의 실제 구현: 스프링의 `ApplicationEventPublisher`(발행) + `@EventListener`(핸들러) 활용 
기본은 동기지만 @Async를 붙이면 비동기가 되고, `@TransactionalEventListener(특히 AFTER_COMMIT)`를 쓰면 "주문 취소 트랜잭션이 커밋된 뒤에만 환불 이벤트를 처리"하도록 시점을 제어할 수 있다.
- 외부 시스템 연동(Kafka 발행 등)은 보통 이 AFTER_COMMIT 핸들러 안에서 한다 — 커밋 전에 이벤트를 외부로 내보냈다가 롤백되면 "취소 안 됐는데 환불된" 사고가 나기 때문이다.

## 10.3 이벤트, 핸들러, 디스패처 구현

**실제 코드 구조 구현 예시 **

스프링을 쓰면 디스패처를 직접 만들 필요 없이 프레임워크가 제공하는 걸 그대로 쓸 수 있다. 

- 이벤트 클래스
발생한 사실을 담는 객체. 과거 시제로 이름 짓고, 핸들러가 후속 작업에 필요한 최소 데이터만 담는다.
```kotlin
data class OrderCanceledEvent(
    val orderNumber: String
)
```
이벤트는 보통 공통 정보(발생 시각 등)를 위한 상위 클래스를 두기도 한다. "무슨 일이, 언제, 어떤 대상에"를 표현하되 과하게 담지 않는 게 핵심이다.

- 디스패처 

스프링이 디스패처 역할을 한다. 직접 구현하지 않고 `ApplicationEventPublisher`를 주입받아 `publishEvent()`를 호출하면 된다.

```kotlin
class CancelOrderService(
    private val eventPublisher: ApplicationEventPublisher
) {
    fun cancel(orderNumber: String) {
        val order = findOrder(orderNumber)
        order.cancel()                                  // 도메인 로직
        eventPublisher.publishEvent(                    // 이벤트 발행
            OrderCanceledEvent(orderNumber)
        )
    }
}
```

도메인 로직 실행 후 이벤트를 발행만 하고 끝낸다. 누가 받는지는 신경 쓰지 않는다.

- 이벤트 핸들러 

이벤트에 반응하는 쪽. `@EventListener`를 붙이면 해당 타입의 이벤트가 발행될 때 자동 호출된다.

```kotlin
@Service
class OrderCanceledHandler(
    private val refundService: RefundService
) {
    @EventListener(OrderCanceledEvent::class)
    fun handle(event: OrderCanceledEvent) {
        refundService.refund(event.orderNumber) // 후속 작업
    }
}
```


![](https://velog.velcdn.com/images/cksgodl/post/e5cb4e1d-7e3c-455f-b8bd-dbc41f4928c1/image.png)



핸들러는 여러 개 둘 수 있다. 알림, 통계 핸들러를 추가해도 발행 코드(CancelOrderService)는 손대지 않는다.

> 이벤트 생성 위치 — 도메인 객체 vs 서비스: 
> 위 예시는 서비스(CancelOrderService)에서 이벤트를 발행했지만, 더 DDD다운 방식은 애그리거트(도메인 객체) 안에서 이벤트를 등록하고 서비스/리포지토리가 이를 꺼내 발행하는 것이다. 
>
> 스프링 데이터의 `AbstractAggregateRoot`와 `@DomainEvents` 애너테이션이 정확히 이 패턴을 지원합니다 — 애그리거트가 `registerEvent()`로 이벤트를 쌓아두면 저장 시점에 자동 발행

1. 애그리거트 — AbstractAggregateRoot 상속
```kotlin
@Entity
class Order(
    @Id val orderNumber: String,
    @Enumerated(EnumType.STRING) var status: OrderStatus
) : AbstractAggregateRoot<Order>() {

    fun cancel() {
        verifyCancellable()           // 취소 가능한지 도메인 규칙 검증
        this.status = OrderStatus.CANCELED
        registerEvent(                // ← 이벤트를 "등록"만 함 (발행 아님)
            OrderCanceledEvent(orderNumber)
        )
    }

    private fun verifyCancellable() {
        if (status == OrderStatus.SHIPPED)
            throw IllegalStateException("이미 배송된 주문은 취소 불가")
    }
}
```

핵심은 `registerEvent()`. 이벤트 발행이 아니라 애그리거트 내부 리스트에 이벤트를 모아두는 동작
`AbstractAggregateRoot`가 이 리스트와 `@DomainEvents / @AfterDomainEventPublication` 처리를 내부적으로 제공한다.

"언제 이벤트가 생기는가(취소 규칙 통과 시)"라는 도메인 규칙이 도메인 객체 안에 들어있는 게 포인트

2. 서비스 — 이제 발행 코드가 없다
```kotlin
@Service
class CancelOrderService(
    private val orderRepository: OrderRepository
) {
    @Transactional
    fun cancel(orderNumber: String) {
        val order = orderRepository.findById(orderNumber)
            .orElseThrow { OrderNotFoundException(orderNumber) }
        order.cancel()                    // 도메인 로직 (이벤트는 내부에 쌓임)
        orderRepository.save(order)       // ← 저장 시점에 이벤트 자동 발행
    }
}
```

3. 핸들러 — 그대로

```kotlin
@Service
class OrderCanceledHandler(
    private val refundService: RefundService
) {
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    fun handle(event: OrderCanceledEvent) {
        refundService.refund(event.orderNumber)
    }
}
```

## 10.4 동기 이벤트 처리 문제

동기 이벤트가 왜 문제를 다 못 푸는가

`ApplicationEventPublisher.publishEvent()`(또는 `@DomainEvents` 자동 발행)는 기본적으로 같은 스레드, 같은 트랜잭션에서 핸들러를 호출. 

즉 발행자가 핸들러 실행이 끝날 때까지 기다립니다. 코드상으로는 발행/구독으로 분리됐지만, 실행 흐름은 여전히 한 줄로 이어진 동기 호출이다.

- `order.cancel() → publishEvent() → [환불 핸들러 실행 완료까지 대기] → cancel()` 리턴
이 구조 때문에 10.1의 직접 호출과 실질적으로 같은 두 문제가 남는다.

- 성능 (응답 지연): 환불 핸들러가 외부 PG를 호출하느라 3초 걸리면, 주문 취소 응답도 3초 늦어진다.
- 트랜잭션 / 장애 전파: 핸들러가 같은 트랜잭션에서 돌면, 환불 핸들러에서 예외가 터질 때 주문 취소까지 롤백된다.

## 10.5 비동기 이벤트 처리

**"동기로는 성능 지연·장애 전파가 안 풀린다" -> 이번에는 그 해법인 비동기 처리를 다룬다. 
**

후속 작업은 "최대한 빨리"가 아니라 "언젠가 반드시"만 보장되면 충분하다.

### 비동기 구현의 네 가지 방식

- 로컬 핸들러 비동기 실행 (@Async): 가장 간단. 핸들러를 별도 스레드 풀에서 실행해 발행자가 기다리지 않게 한다. 같은 프로세스 안에서 동작.
- 메시지 큐 (Kafka, RabbitMQ): 이벤트를 브로커에 넣고 별도 컨슈머가 처리. 프로세스·서버를 넘어 분리되고, 브로커가 이벤트를 보관하므로 유실에 강하다.
- 이벤트 저장소 + 포워더 (Forwarder): 이벤트를 DB(이벤트 저장소)에 저장하고, 별도 포워더가 주기적으로 읽어 핸들러/외부로 전달한다. 포워더가 가져가는 pull 방식.
- 이벤트 저장소 + API: 이벤트를 저장소에 쌓아두고 외부 시스템이 API로 조회해 가져간다. 구독자가 능동적으로 가져가는 방식.

## 10.6 이벤트 적용 시 추가 고려 사항

**이벤트를 실제 시스템에 적용할 때 부딪히는 현실적인 문제**

1. 이벤트 소스(발생 주체)를 정보에 담을지
이벤트에 "누가/무엇이 이 이벤트를 발생시켰는지" 정보를 포함할지 결정해야 한다. 예를 들어 특정 종류의 이벤트만 골라 처리하거나, 발생 출처별로 다르게 반응해야 할 때 필요하다.

2. 포워더 재전송 기준 — 마지막 처리 위치 추적
이벤트 저장소 + 포워더 방식에서, 포워더가 "어디까지 전달했는지"를 기억해야 한다. 마지막으로 처리한 이벤트의 위치(오프셋·ID)를 저장해두고, 거기서부터 다시 읽는다. 

3. 이벤트 유실 — 어디서 잃어버릴 수 있나
이벤트가 유실되는 지점은 크게 둘입니다. (a) 발행 시점: 로컬 핸들러(@Async)는 앱이 죽으면 메모리상의 이벤트가 사라진다. (b) 처리 시점: 핸들러가 이벤트를 받았지만 처리에 실패하는 경우. 유실을 막으려면 이벤트를 DB에 먼저 저장(도메인 데이터와 같은 트랜잭션으로)한 뒤 포워더로 전달하는 방식이 안전하다.

4. 이벤트 처리 순서 / 멱등성 (중복 처리)
비동기·분산 환경에서는 같은 이벤트가 두 번 이상 전달될 수 있다(at-least-once). 포워더가 재시작되며 재전송하거나, 네트워크 재시도로 중복이 생긴다. 
그래서 핸들러는 멱등성(idempotency) 을 갖춰야 한다 — 같은 이벤트를 여러 번 받아도 결과가 한 번 처리한 것과 같아야 한다.
