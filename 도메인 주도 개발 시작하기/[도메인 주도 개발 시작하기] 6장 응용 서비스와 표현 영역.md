## 6.1 표현 영역과 응용 영역

![](https://velog.velcdn.com/images/cksgodl/post/e1669da1-0e5f-416f-a8d1-abffb85a364e/image.png)

### 표현 영역의 책임
표현 영역은 사용자와 가장 가까이 있는 계층

- 사용자가 시스템을 사용할 수 있는 흐름(화면)을 제공하고 제어
- 사용자의 요청을 알맞은 응용 서비스에 전달하고 결과를 사용자에게 제공
- 사용자의 세션을 관리


## 6.2 응용 서비스의 역할

![](https://velog.velcdn.com/images/cksgodl/post/737f64d1-17ac-48d3-af24-0e84e068990f/image.png)


### 응용 서비스의 핵심 역할

- 사용자(클라이언트)의 요청을 처리하기 위해 리포지터리에서 도메인 객체를 가져와 사용
- 도메인 객체 간의 흐름을 제어하는 것이 주된 일
- **도메인 영역과 표현 영역을 연결해주는 창구(facade) 역할**
- **도메인 영역 침입 금지**

## 6.3 응용 서비스의 구현

#### 응용 서비스의 크기

응용 서비스를 구현할 때 가장 먼저 결정할 것은 클래스의 크기(범위)이다.

![](https://velog.velcdn.com/images/cksgodl/post/9dd07f4c-82bd-4260-ad93-855927542f0c/image.png)

#### 응용 서비스의 인터페이스와 클래스

일반적으로 인터페이스 분리는 필요 없음
응용 서비스의 구현 클래스가 하나만 존재하는 경우가 대부분
인터페이스는 다음과 같은 경우에만 도입

- 구현 클래스가 여러 개일 때 (런타임 교체 등)
- 테스트를 위한 mock 구현이 필요한데 mock 라이브러리로 처리하기 어려울 때


#### 메서드 파라미터와 값 리턴

파라미터는 별도 데이터 클래스(Command/Request 객체) 로 전달
표현 영역의 객체(예: Spring Form)를 그대로 받지 말 것 → 응용 영역이 표현 영역에 의존하게 됨

```kotlin
data class ChangePasswordCommand(
    val memberId: String,
    val currentPassword: String,
    val newPassword: String
)

@Service
class ChangePasswordService(
    private val memberRepository: MemberRepository
) {
    @Transactional
    fun changePassword(command: ChangePasswordCommand) {
        ...
}
```

표현 영역에 의존하지 않기

#### 트랜잭션 처리

트랜잭션 경계는 응용 서비스에서 시작하고 끝남
Spring에서는 @Transactional 어노테이션으로 선언적 트랜잭션 처리
도메인 상태 변경은 트랜잭션 내에서 일관되게 반영되어야 함

## 6.4 표현 영역

#### 표현 영역의 책임

사용자가 시스템을 사용할 수 있는 흐름(화면)을 제공하고 제어
사용자의 요청을 알맞은 응용 서비스에 전달하고 결과를 사용자에게 응답
사용자의 세션을 관리

![](https://velog.velcdn.com/images/cksgodl/post/9f923ab2-fc0d-472d-9aab-ac1c71591fc0/image.png)

#### 표현 영역이 해야 할 일

- 사용자 입력 값을 응용 서비스가 요구하는 형식의 Command 객체로 변환
- 응용 서비스 결과를 사용자에게 알맞은 형식(View 이름, JSON, HTML 등)으로 변환
- 응용 서비스에서 발생한 예외를 적절한 응답으로 처리

#### 표현 영역의 검증

표현 영역에서는 요청 데이터 형식 검증까지만 담당
@NotBlank, @Email, @Size 형식 검증만

비즈니스 규칙 검증은 도메인의 몫

## 6.5 값 검증

값 검증은 표현 영역과 응용 서비스 두 곳에서 모두 수행

![](https://velog.velcdn.com/images/cksgodl/post/3b5390d8-64d4-4856-99cd-8e1179f27a43/image.png)

#### 표현 영역의 검증

- 입력 폼 데이터의 형식 검증 담당
- Spring의 @Valid, Bean Validation 어노테이션 활용

#### 응용 서비스의 검증

- 논리적 검증 담당 (DB 조회 등 외부 자원이 필요한 검증)
- 표현 영역에서 잡을 수 없는 검증
   - ID 중복 여부
   - 참조하는 회원/상품이 실제 존재하는지
   - 도메인 규칙 위반 여부
   
#### validator 활용

Spring에서 제공하는 Validator 인터페이스로 검증 로직을 별도 클래스로 분리 가능 -> 컨트롤러 코드에서 검증 로직이 분리

![](https://velog.velcdn.com/images/cksgodl/post/53e7dc5b-0a89-4ad1-bf9e-50222bf22f34/image.png)
